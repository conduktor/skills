# Encrypt Kafka data with Conduktor Gateway

## Agent workflow

1. Run `conduktor get Interceptor --gateway -o yaml` to check existing encryption interceptors
2. Ask what to encrypt: specific fields (field-level) or entire payload
3. Ask which KMS provider (Vault, AWS KMS, Azure, GCP, or in-memory for dev)
4. Ask which topics and whether to scope by virtual cluster, service account, or group
5. Run `conduktor get VirtualCluster -o name` and `conduktor get GatewayServiceAccount -o name` to get real scope targets
6. Generate the complete `Interceptor` YAML with correct `pluginClass`, KMS config, and scope targeting
7. Generate the matching `DecryptPlugin` interceptor for consumers
8. Show both YAMLs and offer to run `conduktor apply -f --dry-run`
9. On approval, run `conduktor apply -f`

Gateway encrypts data as it passes through the proxy, before it reaches the broker. Unlike TLS, data stays encrypted at rest on Kafka brokers. Requires Conduktor Shield license.

## When to use this

- Encrypt PII/sensitive fields before they enter Kafka (passwords, credit cards, emails).
- Full payload encryption when the entire message is sensitive.
- Field-level encryption when only specific fields need protection (Avro, Protobuf, JSON).
- Crypto shredding: use mustache-templated `keySecretId` so deleting the key makes data unrecoverable.
- Encrypt on produce (recommended) or on consume (deprecated since Gateway v3.16.0).

## How it works (envelope encryption)

Gateway uses envelope encryption. The data is never encrypted directly with the KMS key.

| Term | Definition |
|------|-----------|
| **KMS** | Key Management Service storing the KEK. Gateway never sees the KEK directly. |
| **KEK** | Key Encryption Key in the KMS. Encrypts/decrypts the DEK. |
| **DEK** | Data Encryption Key generated by Gateway. Encrypts actual data. |
| **EDEK** | Encrypted DEK. Stored alongside encrypted data in Kafka record headers. |

**Encrypt flow**: Gateway generates DEK -> sends DEK to KMS -> KMS encrypts DEK with KEK, returns EDEK -> Gateway caches DEK+EDEK (configurable TTL) -> encrypts data with DEK -> stores EDEK in record headers -> sends to Kafka.

**Decrypt flow**: Gateway reads EDEK from headers -> sends EDEK to KMS -> KMS decrypts with KEK, returns DEK -> Gateway decrypts data with DEK.

DEK caching reduces KMS calls. Cache TTL is configurable.

## Encryption modes

### Full payload encryption

Encrypts the entire record value, key, or headers. Works on structured and unstructured messages.

```yaml
apiVersion: gateway/v2
kind: Interceptor
metadata:
  name: encrypt-full-payload
spec:
  pluginClass: io.conduktor.gateway.interceptor.EncryptPlugin
  priority: 100
  config:
    topic: "sensitive-.*"
    kmsConfig:
      vault:
        uri: http://vault:8200
        token: ${VAULT_TOKEN}
        version: 1
    recordValue:
      payload:
        keySecretId: vault-kms://vault:8200/transit/keys/full-payload-key
        algorithm: AES128_GCM
```

### Field-level encryption (list-based)

Specify exact fields to encrypt in the interceptor config. Fields support dot notation for nested structures.

```yaml
apiVersion: gateway/v2
kind: Interceptor
metadata:
  name: encrypt-fields
spec:
  pluginClass: io.conduktor.gateway.interceptor.EncryptPlugin
  priority: 100
  config:
    topic: ".*"
    schemaRegistryConfig:
      host: http://schema-registry:8081
    kmsConfig:
      vault:
        uri: http://vault:8200
        token: ${VAULT_TOKEN}
        version: 1
    recordValue:
      fields:
        - fieldName: password
          keySecretId: vault-kms://vault:8200/transit/keys/password-secret
          algorithm: AES128_GCM
        - fieldName: address.zipCode
          keySecretId: vault-kms://vault:8200/transit/keys/address-secret
          algorithm: AES256_GCM
```

### Field-level encryption (schema-based)

Fields are encrypted based on annotations in the schema itself (`conduktor.keySecretId`, `conduktor.algorithm`, `conduktor.tags`). No field list in the interceptor config.

```yaml
apiVersion: gateway/v2
kind: Interceptor
metadata:
  name: encrypt-schema-based
spec:
  pluginClass: io.conduktor.gateway.interceptor.EncryptSchemaBasedPlugin
  priority: 100
  config:
    topic: ".*"
    schemaRegistryConfig:
      host: http://schema-registry:8081
    kmsConfig:
      vault:
        uri: http://vault:8200
        token: ${VAULT_TOKEN}
        version: 1
    defaultKeySecretId: vault-kms://vault:8200/transit/keys/default-key
    defaultAlgorithm: AES128_GCM
    tags:
      - PII
      - ENCRYPTION
    namespace: "conduktor."
```

In the schema, annotate fields with `conduktor.keySecretId`, `conduktor.algorithm`, or `conduktor.tags`. Fields matching any of these (or a tag in the interceptor's `tags` list) get encrypted. Unmatched fields are left untouched. Works with JSON Schema, Avro, and Protobuf.

## Step by step

### 1. Choose a KMS

| KMS | `keySecretId` prefix | Example |
|-----|---------------------|---------|
| Vault Transit | `vault-kms://` | `vault-kms://vault:8200/transit/keys/my-key` |
| AWS KMS | `aws-kms://` | `aws-kms://arn:aws:kms:us-east-1:123456789012:key/my-key` |
| Azure Key Vault | `azure-kms://` | `azure-kms://https://my-vault.vault.azure.net/keys/my-key/version` |
| GCP KMS | `gcp-kms://` | `gcp-kms://projects/p/locations/l/keyRings/r/cryptoKeys/k` |
| In-memory | `in-memory-kms://` | `in-memory-kms://my-key` (dev only) |

Configure the connection in `kmsConfig` under `spec.config`. Multiple KMS providers can coexist in the same interceptor.

### 2. Create encryption interceptor

Pick the plugin class based on your mode:

| Mode | On produce | On consume (deprecated) |
|------|-----------|------------------------|
| List-based | `io.conduktor.gateway.interceptor.EncryptPlugin` | `io.conduktor.gateway.interceptor.FetchEncryptPlugin` |
| Schema-based | `io.conduktor.gateway.interceptor.EncryptSchemaBasedPlugin` | `io.conduktor.gateway.interceptor.FetchEncryptSchemaBasedPlugin` |

### 3. Create decryption interceptor

Plugin class: `io.conduktor.gateway.interceptor.DecryptPlugin`

Decrypt all encrypted fields (no field list needed):
```yaml
apiVersion: gateway/v2
kind: Interceptor
metadata:
  name: decrypt-all
spec:
  pluginClass: io.conduktor.gateway.interceptor.DecryptPlugin
  priority: 100
  config:
    topic: ".*"
    schemaRegistryConfig:
      host: http://schema-registry:8081
    kmsConfig:
      vault:
        uri: http://vault:8200
        token: ${VAULT_TOKEN}
        version: 1
```

Decrypt specific fields only:
```yaml
config:
  topic: ".*"
  kmsConfig:
    vault:
      uri: http://vault:8200
      token: ${VAULT_TOKEN}
  recordValueFields:
    - password
    - address.zipCode
  recordKeyFields:
    - bank.accountNo
```

### 4. Apply via CLI

```bash
conduktor apply -f encrypt-interceptor.yaml
conduktor apply -f decrypt-interceptor.yaml
```

## Configuration reference

### Encryption (common)

| Key | Type | Default | Notes |
|-----|------|---------|-------|
| `topic` | String | `.*` | Regex for topic matching |
| `schemaRegistryConfig.host` | String | | Required for Avro/JSON/Protobuf |
| `kmsConfig` | Object | | One or multiple KMS connections |
| `schemaDataMode` | String | `preserve_avro` | `preserve_avro` or `convert_json` |
| `compressionType` | Enum | `none` | `none`, `gzip`, `snappy`, `lz4`, `zstd` (full payload only) |
| `errorPolicy` | String | `fail_on_encrypted` | `fail_on_encrypted` or `skip_already_encrypted` |

### List-based fields (`recordValue` / `recordKey`)

| Key | Type | Notes |
|-----|------|-------|
| `payload.keySecretId` | String | Full payload encryption. Supports mustache templates. |
| `payload.algorithm` | Algorithm | Default `AES128_GCM` |
| `fields[].fieldName` | String | Dot notation for nested fields, `[0]` for arrays |
| `fields[].keySecretId` | String | Supports mustache templates |
| `fields[].algorithm` | Algorithm | |

### Schema-based

| Key | Type | Default | Notes |
|-----|------|---------|-------|
| `defaultKeySecretId` | String | | Fallback when schema has no `keySecretId` |
| `defaultAlgorithm` | Algorithm | `AES128_GCM` | Fallback algorithm |
| `tags` | List[String] | | Tags to match in schema annotations |
| `namespace` | String | `conduktor.` | Schema annotation prefix |

### Algorithms

`AES128_GCM` (default), `AES256_GCM`, `AES128_EAX`, `AES256_EAX`, `AES128_CTR_HMAC_SHA256`, `AES256_CTR_HMAC_SHA256`, `CHACHA20_POLY1305`, `XCHACHA20_POLY1305`

### Mustache templates for `keySecretId`

`{{record.topic}}`, `{{record.key}}`, `{{record.value.fieldName}}`, `{{record.header.headerName}}`, `{{record.value.list[0].field}}`

Useful for crypto shredding: one key per user, delete the key to make their data unrecoverable.

## Key rotation

**KEK rotation** is your responsibility. Perform it through your KMS provider. Gateway automatically uses the new KEK version for new DEKs. Old EDEKs remain decryptable as long as the old KEK version exists in the KMS.

**DEK limits**: AES-GCM has a ~2^32 message limit per DEK before nonce collision risk. For high-traffic topics, use `XCHACHA20_POLY1305` (~2^80 limit) or rotate keys via unique `keySecretId` values.

## Common mistakes

- **Missing `schemaRegistryConfig`**: required when encrypting Avro, JSON Schema, or Protobuf messages. Without it, Gateway cannot parse structured fields.
- **Using `in-memory-kms://` in production**: keys are lost on Gateway restart. Use Vault, AWS, Azure, or GCP.
- **Encrypt and decrypt on different KMS**: the decrypt interceptor must reach the same KMS that encrypted the data.
- **Forgetting `kmsConfig` in the decrypt interceptor**: DecryptPlugin also needs KMS access to decrypt the EDEK.
- **Using `FetchEncryptPlugin`/`FetchEncryptSchemaBasedPlugin` for new setups**: on-consume encryption is deprecated since v3.16.0. Use on-produce plugins.
- **Mustache template referencing encrypted field**: if `fieldName: email` and `keySecretId` uses `{{record.value.email}}`, decryption becomes impossible. The templated field must not be in the encrypted field list.
- **AES-GCM on high-throughput topics without rotation**: a single DEK can safely encrypt ~4 billion messages with AES-GCM. Beyond that, switch algorithm or rotate keys.
- **Environment variable syntax in YAML**: use `${VAR}` for values Gateway resolves at startup. Do not confuse with mustache `{{...}}` which resolves per record.
